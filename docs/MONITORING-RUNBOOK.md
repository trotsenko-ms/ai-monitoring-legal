## 3. Щотижневий огляд

**Що перевіряти щотижня (кожного понеділка):**

1. **Стабільність**: скільки запусків за тиждень завершилися успішно vs. з помилкою
   - Ціль: ≥ 95% successful runs (max 1 збій на тиждень)
2. **Precision** (вибіркова перевірка вручну 10–20% знайдених НПА):
   - Чи всі знайдені документи дійсно релевантні?
   - Ціль: ≥ 85%
3. **Recall** (перехресна перевірка з офіційними джерелами):
   - Перевірити вручну 2–3 джерела на наявність НПА, яких агент не знайшов
   - Ціль: ≥ 80%
4. **Дедуплікація**: переглянути `data/dedup/*.json` — чи є підозрілі записи?
5. **Tokens**: середнє споживання токенів на запуск (з `runs/` → `stats.tokens_used`)

## 4. Дії при аномаліях

### 4.1 Red run (workflow failed)

1. Відкрити GitHub Actions → знайти failed job → переглянути logs
2. Типові причини:
   - `ANTHROPIC_API_KEY` прострочений або відсутній → оновити в Secrets
   - `GMAIL_APP_PASSWORD` відсутній → оновити в Secrets
   - Джерело недоступне (timeout) → тимчасово, повторний запуск через `workflow_dispatch`
   - Помилка парсингу (сайт змінив структуру) → оновити prompt відповідного агента
3. Після виправлення: запустити вручну через `workflow_dispatch`

### 4.2 Yellow run (succeeded з помилками в артефакті)

1. Переглянути `stats.errors` у відповідному `runs/*.json`
2. `source_unavailable` — тимчасово, моніторити наступний запуск
3. `validation_failed` > 20% items — перевірити prompt validator-агента

### 4.3 Нульовий результат (0 items) протягом 3+ запусків поспіль

1. Перевірити джерело вручну (чи є там нові документи?)
2. Якщо джерело оновлювалось — проблема в агенті → уточнити пошукові параметри в prompt

### 4.4 Забагато false positives (precision < 85%)

1. Переглянути prompt відповідного Monitor — уточнити критерії фільтрації
2. Додати до prompt приклади нерелевантних документів
