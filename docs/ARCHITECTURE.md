## 1. Загальний опис

**Назва проєкту:** ШІ-Агенти моніторингу законодавства України

**Мета:** автоматизоване відстеження змін у нормативно-правових актах України через систему спеціалізованих ШІ-агентів з валідацією результатів і доставкою дайджестів на email.

**Користувач:** юрист, що потребує оперативної інформації про зміни законодавства без щоденного ручного моніторингу.

**Поточний статус:** пілотний проєкт. Тестується на двох напрямках моніторингу.

## 2. Пілотний статус

### 2.1 Що означає «пілот»

Проєкт перебуває у фазі експериментальної перевірки гіпотез про:
- працездатність мульти-агентної архітектури з валідаційним шаром;
- якість і повноту автоматизованого моніторингу через web search;
- ефективність pipeline-патерну (Monitor → Orchestrator → Validator);
- доцільність формату доставки результатів через email-дайджести.

### 2.2 Скоуп пілоту

**У межах пілоту тестуємо два напрямки моніторингу:**

1. **Електронний документообіг (ЕДО)** — попередньо розроблений напрямок, мігрує з v1.0 у нову архітектуру.
2. **Зареєстровані законопроекти / опубліковані акти КМУ / опубліковані акти НБУ** — новий напрямок з трьома спеціалізованими monitor-агентами.

**Поза скоупом пілоту (фаза 2):**
- корпус повних текстів діючих НПА (`data/corpus/`);
- Аналізатори (Рівень 2 поглибленого аналізу);
- агент звітності (Рівень 3 за шаблоном);
- розширення на інші сфери права.

### 2.3 Критерії успіху пілоту

Пілот вважається успішним, якщо:
- 4 monitor-агенти стабільно працюють за розкладом протягом 30 днів;
- precision знайдених НПА ≥ 85%, recall ≥ 80%;
- Validator відсіює помилкові спрацювання (false positives) ≥ 90%;
- email-дайджести доставляються двічі на день без збоїв;
- користувач підтверджує практичну корисність формату.

### 3.2 Асиметрія гілок — обґрунтування

Гілки навмисно мають різну форму:

- **ЕДО — лінійний pipeline.** Один Monitor шукає за патернами, один Validator перевіряє. Координувати нічого, тому orchestrator не потрібен. Додавати його заради симетрії — overengineering.
- **Держоргани — pipeline з координатором.** Три Monitor працюють паралельно (різні джерела з різною структурою), їхні результати потрібно зібрати і передати єдиному Validator. Це і є функція `orchestrator-state`.

### 3.3 Компоненти системи

| Тип | Агент | Призначення |
|-----|-------|-------------|
| Monitor | `monitor-edo` | Пошук НПА у сфері електронного документообігу через zakon.rada.gov.ua, kmu.gov.ua, bank.gov.ua за патернами |
| Monitor | `monitor-rada-bills` | Моніторинг зареєстрованих законопроектів через `itd.rada.gov.ua/billinfo/Bills/period` |
| Monitor | `monitor-kmu` | Моніторинг опублікованих актів КМУ через `kmu.gov.ua/npasearch` |
| Monitor | `monitor-nbu` | Моніторинг опублікованих актів НБУ через `bank.gov.ua/ua/legislation` |
| Orchestrator | `orchestrator-state` | Координація трьох держорганних Monitor: паралельний запуск, збір артефактів, передача консолідованого артефакту далі |
| Validator | `validator-edo` | Перевірка результатів monitor-edo: доступність URL, наявність документа, дата публікації, релевантність |
| Validator | `validator-state` | Аналогічна перевірка для консолідованого артефакту orchestrator-state |

### 3.4 Workflow одного циклу (приклад: гілка держорганів)

1. Тригер (cron 08:00 або 17:00, або manual)
2. `orchestrator-state` запускає три monitor-агенти паралельно
3. Кожен Monitor завантажує свій `seen_urls_*.json` для дедуплікації
4. Кожен Monitor виконує пошук на своєму джерелі
5. Кожен Monitor продукує JSON-артефакт за стандартним контрактом
6. `orchestrator-state` збирає три артефакти, об'єднує в консолідований
7. `validator-state` приймає консолідований артефакт, перевіряє кожен item
8. Validator продукує валідований артефакт зі статусами по items
9. Сервіс доставки формує email-дайджест за валідованим артефактом
10. Email надсилається на `legal.monitor.ai@gmail.com`
11. Кожен агент оновлює свій `seen_urls_*.json` (нові URL + видалення >90 днів)
12. Запис фіксується в `runs/{agent_name}/{timestamp}.json`

## 4. Multi-agent pipeline і JSON-контракт

### 4.1 Принцип pipeline

Кожен агент дотримується єдиної схеми життєвого циклу:

```
підготовка → основна робота → внутрішня самоперевірка → передача (handoff)
```

Це робить агентів взаємозамінними та полегшує додавання нових джерел.

### 4.2 JSON-контракт між агентами

Кожен Monitor продукує артефакт у стандартизованому форматі:

```json
{
  "agent_id": "monitor-rada-bills",
  "agent_version": "1.0",
  "run_id": "2026-05-06T08-00-00Z",
  "period": {
    "from": "2026-05-05T17:00:00Z",
    "to": "2026-05-06T08:00:00Z"
  },
  "source_url": "https://itd.rada.gov.ua/billinfo/Bills/period",
  "items": [
    {
      "title": "Проект Закону про ...",
      "number": "10234",
      "date_registered": "2026-05-05",
      "url": "https://itd.rada.gov.ua/billinfo/Bills/Card/...",
      "raw_status": "found"
    }
  ],
  "stats": {
    "items_found": 5,
    "duplicates_filtered": 12,
    "errors": []
  }
}
```

### 4.3 Слабка зчепленість (loose coupling)

Передача між агентами — через файлову систему (artifacts читаються з `/runs/`), а не через прямий програмний виклик. Переваги:
- кожен агент тестується незалежно;
- збій одного агента не каскадує на інших миттєво;
- простіше дебажити (повна історія handoff збережена).

### 5.1 Призначення кожної гілки

**`agents/`** — інструкції (system prompts), конфіги та per-agent README. Один агент = одна папка. Усе що визначає поведінку агента живе тут.

**`shared/`** — спільна інфраструктура. JSON-схеми контрактів між агентами (для валідації та документації), повторювані фрагменти промптів (наприклад, спільний preamble про обмеження).

**`data/`** — операційні дані. Зараз тільки дедуплікація. У фазі 2 додасться `corpus/` для повних текстів діючих НПА.

**`tests/`** — контрольоване тестування. `fixtures/` — статичні приклади вхідних даних; `cases/` — інтенціональні сценарії з очікуваними результатами; `results/` — куди записуються прогони для порівняння.

**`runs/`** — продакшн-аудит. Кожен реальний запуск кожного агента залишає тут JSON-файл з input, output, метриками, помилками. Це історична правда системи.

**`docs/`** — документація для людей. Архітектура, журнал змін, per-agent гайди.

## 6. Розклад і доставка

### 6.1 Розклад запусків

| Час | Тип | Гілка | Тригер |
|-----|-----|-------|--------|
| 08:00 (Київ) | Cron | Держоргани (rada-bills + kmu + nbu → orchestrator → validator) | — |
| 08:30 (Київ) | Cron | ЕДО (monitor-edo → validator-edo) | — |
| 17:00 (Київ) | Cron | Держоргани | — |
| За запитом | Manual | ЕДО | Команда «ЕДО» |
| За запитом | Manual | Держоргани | Команда «НПА» |

Типовий день: 3 запуски cron (08:00 держоргани → 08:30 ЕДО → 17:00 держоргани).

Період пошуку кожної гілки — від попереднього запуску тієї ж гілки до поточного.

### 6.2 Email-доставка

**Адреса доставки:** `legal.monitor.ai@gmail.com` (окрема скринька для дайджестів)

**Окремий дайджест на кожен запуск гілки.** За типовий день надходить 3 листи:

1. **08:00 → Держоргани digest**
   - Тема: `[НПА] Дайджест моніторингу за {від попереднього запуску до 08:00}`
   - Зміст: консолідовано законопроекти ВРУ, акти КМУ, акти НБУ
2. **08:30 → ЕДО digest**
   - Тема: `[ЕДО] Дайджест моніторингу за {від попереднього запуску до 08:30}`
   - Зміст: список НПА у сфері ЕДО, що пройшли валідацію
3. **17:00 → Держоргани digest**
   - Тема: `[НПА] Дайджест моніторингу за {від 08:00 до 17:00}`
   - Зміст: аналогічно п.1

### 6.3 Формат дайджесту

Простий формат для пілоту:

```
Період моніторингу: з {from} до {to}

ЗНАЙДЕНО {N} нових документів:

1. {Назва документа}
   Реквізити: {номер, дата, орган}
   Посилання: {URL на офіційний текст}

2. ...

Якщо нічого нового — текст: «За вказаний період нових змін не виявлено».
```

Розширений формат (з оцінкою релевантності) — фаза 2.

## 7. Дедуплікація

### 7.1 Per-agent files

Кожен Monitor веде власний файл дедуплікації:

```
data/dedup/
├── seen_urls_edo.json
├── seen_urls_rada_bills.json
├── seen_urls_kmu.json
└── seen_urls_nbu.json
```

### 7.2 Формат файлу

```json
{
  "agent_id": "monitor-rada-bills",
  "last_updated": "2026-05-06T08-00-00Z",
  "urls": [
    {
      "url": "https://itd.rada.gov.ua/billinfo/Bills/Card/53245",
      "found_date": "2026-05-01"
    }
  ]
}
```

### 7.3 Логіка очищення

При кожному запуску агент:
1. Завантажує власний seen_urls файл
2. Видаляє записи де `found_date` старше за 90 днів від поточної дати
3. Виконує пошук
4. Фільтрує дублікати (URL вже в списку)
5. Додає нові URL з поточною датою
6. Зберігає оновлений файл

Переваги per-agent підходу:
- незалежне очищення (різний темп оновлення джерел);
- відсутність race conditions при паралельному запуску;
- простіше дебажити кожне джерело окремо.

## 8. Тестування і фіксація запусків

### 8.1 Тестова інфраструктура (`tests/`)

**`tests/fixtures/`** — статичні зразки вхідних даних. Приклади HTML-сторінок з джерел (для тестування парсингу), приклади JSON-відповідей API. Дозволяють тестувати агентів без реальних запитів до зовнішніх сервісів.

**`tests/cases/`** — інтенціональні сценарії. Структура `cases/{agent_name}/{case_id}.json`. Кожен кейс містить input (або посилання на fixture) і expected output. Покриваємо edge cases: порожній результат, дублікат, помилка джерела, нерелевантний документ.

**`tests/results/`** — куди записуються результати прогонів тест-кейсів. Дозволяють порівнювати з expected output і відстежувати регресії.

### 8.2 Продакшн-фіксація (`runs/`)

Кожен реальний запуск зберігається у `runs/{agent_name}/{timestamp}.json` у форматі:

```json
{
  "agent_id": "monitor-rada-bills",
  "run_id": "2026-05-06T08-00-00Z",
  "trigger": "cron",
  "input": { "...": "..." },
  "output": { "...": "..." },
  "stats": {
    "duration_ms": 12450,
    "items_found": 5,
    "duplicates_filtered": 12,
    "tokens_used": 8430,
    "errors": []
  }
}
```

Призначення:
- аудит (що саме знайшов агент конкретного дня);
- дебаг (відтворення проблеми за збереженим input);
- метрики (precision, recall, час виконання за період);
- fallback (повторна доставка дайджесту з історичних даних).

- Reporter (Рівень 3): фінальні звіти за шаблоном.
- Розширення на інші сфери права (податки, праця, корпоративне).

## 11. Метрики

### 11.1 Метрики агентів

| Метрика | Опис | Цільове значення (пілот) |
|---------|------|--------------------------|
| Precision | % релевантних серед знайдених | ≥ 85% |
| Recall | % знайдених від реально опублікованих | ≥ 80% |
| False positives | НПА, помилково включені | ≤ 10% |
| False negatives | Пропущені важливі НПА | ≤ 5% |
| Validator filter rate | % відсіяних валідатором | моніториться |
| Час виконання monitor | Середній час одного запуску | < 3 хв |
| Час виконання повного pipeline | Tригер → email | < 8 хв |

### 11.2 Метрики проєкту

- Кількість агентів у продакшені (зараз: 0, ціль пілоту: 7)
- Покриття напрямків (поточно: 2 — ЕДО + держоргани)
- Стабільність розкладу (% запусків без збоїв)
- Якість документації (повнота, актуальність)

## 14. Адміністрування документу

### 14.1 Версіонування

Цей документ зберігається у `docs/ARCHITECTURE.md`. Версія оновлюється при:
- зміні архітектури (нові компоненти, зміна workflow);
- додаванні нового агента;
- інтеграції нового сервісу або джерела;
- зміні roadmap.

### 14.2 Формат версій

`v[major].[minor]` — наприклад v2.0, v2.1, v3.0.
- **major** — кардинальні зміни (нова архітектура, фазовий перехід);
- **minor** — додавання компонентів, уточнення параметрів.

### 14.3 Журнал змін

Окремий файл `docs/CHANGELOG.md` фіксує всі зміни.
